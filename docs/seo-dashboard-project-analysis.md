> **Document created by Kiro after full repository inspection — May 2026**

## 1. Executive Summary

### What the product is

The CIM SEO Automation Platform is a Python-based, server-side SEO reporting and monitoring system built for the **Chartered Institute of Mining (CIM)**, a Canadian professional association. It is not a web application or interactive dashboard in the browser sense. It is a **pipeline of Python scripts** that pull data from external APIs, process it, generate charts, assemble HTML/PDF reports, and push those reports to Monday.com and Google Sheets. Reports are also deployed to GitHub Pages as static HTML files.

### Who it is for

The system is built for an internal SEO/digital team at CIM. Reports are consumed by:

- SEO analysts running the pipelines
- Digital marketing managers reviewing weekly performance
- Stakeholders and executives receiving Monday.com updates

There is no self-service interface. All reports are generated by running scripts (manually or via GitHub Actions cron jobs).

### What SEO/reporting problems it currently solves

- Automates weekly GSC and GA4 data pulls and comparison reports
- Tracks a defined list of 25 branded and non-branded keywords
- Monitors Core Web Vitals and PageSpeed scores for tracked pages
- Crawls the CIM website for broken links and internal linking gaps
- Scores content pages for refresh/archive recommendations
- Verifies pages for AI snippet readiness (hallucination risk)
- Generates AI-written executive bullet summaries via Groq/Llama
- Posts reports to Monday.com and logs historical data to Google Sheets
- Produces a monthly master dashboard combining all data sources

### What value it currently provides

- Saves significant manual reporting time (all data pulls are automated)
- Provides week-over-week and month-over-month comparisons
- Surfaces CTR opportunity gaps (page-1 queries with low CTR)
- Flags broken links and redirect chains
- Identifies orphan pages and weak internal link targets
- Provides AI-generated narrative summaries for stakeholder communication

### What value it does NOT yet provide

- No real-time or interactive dashboard (all reports are static HTML/PDF)
- No conversion or revenue attribution
- No competitor monitoring or backlink analysis
- No search intent classification or topic cluster analysis
- No branded vs non-branded traffic split (partially attempted via regex)
- No geographic or device-level SEO strategy
- No content decay trend detection over time
- No cannibalization detection
- No client-facing or white-label reporting capability
- No alerting system (no email/Slack notifications on anomalies)

## 2. Current Project Structure

### Plain-English Overview

The project lives entirely in a single flat directory — there are no subdirectories for source code. All Python scripts, CSV config files, and documentation markdown files sit at the root level. The only subdirectories are `.github/workflows/` (CI/CD), `.kiro/steering/` (Kiro AI context), `docs/` (this document), and `.git/`. Output files (charts, reports, CSVs) are written to the root directory or to `charts/`, `reports/`, `screenshots/`, and `evidence/` subdirectories created at runtime.

There is no frontend framework, no database, no web server, and no API layer. The "dashboard" is a self-contained HTML file with base64-embedded chart images, generated by Python and either opened locally or deployed to GitHub Pages.

### Architecture Pattern

```
GitHub Actions (cron/manual trigger)
        |
master_orchestrator.py  ←→  monthly_master_report.py
        |                           |
  [10 sub-scripts run in parallel/serial]
        |
  [CSV outputs written to disk]
        |
  [Charts generated via matplotlib]
        |
  [HTML assembled with embedded charts]
        |
  [Monday.com upload]  +  [Google Sheets append]
        |
  [GitHub Pages deploy]
```

## 4. Current Reporting Functionality

### Does the project generate reports?
Yes. Every script produces at least one HTML report. Most also produce CSV data files and a Markdown summary. PDF generation was removed (WeasyPrint dependency dropped per `requirements.txt` comments) — all reports are now self-contained HTML with base64-embedded charts.

### Are reports static or dynamic?
Static. Reports are generated at run time and represent a snapshot of the data at that moment. There is no live data refresh, no user-triggered query, and no interactive filtering.

### Are reports exportable?
The HTML files are self-contained (charts embedded as base64) so they can be shared as single files. They are also attached to Monday.com updates. There is no PDF export currently (removed), no Excel export, and no CSV download from the dashboard UI.

### Are there charts, summaries, insights, or recommendations?
Yes — all four:
- **Charts**: matplotlib PNG charts embedded in HTML
- **Summaries**: AI-generated executive bullet points (Groq/Llama 3.3-70b)
- **Insights**: Deterministic bullets computed from data (always present even if AI fails)
- **Recommendations**: Content audit recommends refresh/archive; internal linking suggests link opportunities; AI snippet verification gives per-page recommendations

### Does it support monthly/quarterly comparisons?
Weekly: week-over-week (7 days vs prior 7 days). Monthly: month-over-month (previous full calendar month vs month before that). No quarterly comparison exists. No year-over-year comparison exists.

### Does it support client-ready or stakeholder-ready reporting?
Partially. The HTML design is polished (Playfair Display typography, brand colors, inverted KPI cards). The monthly dashboard has a table of contents and 10 sections. However:
- Reports require the recipient to open an HTML file or click a Monday.com attachment
- There is no email delivery
- There is no PDF (removed)
- There is no white-label or client-configurable version

### Is the reporting descriptive only, or does it provide interpretation?
Both. Deterministic bullets describe what happened. AI bullets interpret why and what to do. The content audit provides a scored recommendation. The CTR opportunity charts identify specific actionable gaps.

## 11. Technical Architecture Review

### Frameworks and Libraries

- **Language**: Python 3.11 (GitHub Actions), 3.8+ minimum
- **Data processing**: pandas, numpy
- **Visualization**: matplotlib, seaborn (seaborn imported in requirements but not heavily used in code)
- **Web scraping**: BeautifulSoup4, aiohttp, requests, Playwright (Chromium)
- **Google APIs**: google-api-python-client, google-auth, google-analytics-data, gspread
- **AI/LLM**: openai SDK pointed at Groq endpoint (llama-3.3-70b-versatile)
- **Async**: asyncio, aiohttp for concurrent HTTP, ThreadPoolExecutor for parallel API calls
- **HTML generation**: Pure Python string concatenation (no templating engine)
- **PDF**: Removed (WeasyPrint and ReportLab both dropped)
- **Scheduling**: GitHub Actions cron jobs

### Architecture Observations

| Technical Area | Current State | Risk/Issue | Improvement Opportunity |
|---|---|---|---|
| Code organization | Flat directory, all scripts at root | Hard to navigate as project grows; no separation of concerns | Introduce `src/`, `reports/`, `utils/`, `config/` subdirectory structure |
| HTML generation | Pure Python f-string concatenation, thousands of lines of inline HTML | Extremely fragile; HTML errors are invisible at runtime; no separation of logic and presentation | Introduce Jinja2 templating |
| State management | None — each script run is stateless; CSVs written to disk are the only persistence | Scripts depend on CSVs from previous scripts; if one fails, downstream scripts get empty DataFrames | Add explicit dependency checking and error propagation |
| Data persistence | Google Sheets (append-only) + local CSVs (overwritten each run) | No queryable historical database; Google Sheets has row limits and API rate limits | Add SQLite or PostgreSQL for local historical storage |
| Authentication | Google Service Account key file (`gsc-key.json`) written from GitHub Secret at runtime | Key file written to disk in CI; not encrypted at rest during run | Use Workload Identity Federation for keyless auth in GitHub Actions |
| API error handling | Try/except returning empty DataFrames on failure | Silent failures — a failed API call produces an empty chart with "No data available" message, not an error | Add explicit error reporting, distinguish between "no data" and "API failure" |
| Parallelism | ThreadPoolExecutor (8 workers) for API calls; asyncio for crawlers | Thread-safe service objects correctly implemented; good pattern | Consider asyncio throughout for consistency |
| Chart generation | matplotlib with Agg backend, saved to PNG files | Charts are static images; no interactivity | Consider Plotly for interactive HTML charts |
| Report assembly | Charts embedded as base64 in HTML | Files can be 5-15MB; slow to open in browser | Lazy-load charts or serve from CDN |
| Testing | No tests exist | Zero test coverage; regressions are invisible | Add pytest with fixtures for data processing functions |
| Configuration | Mix of environment variables, CSV files, and hardcoded values | GSC property and GA4 property ID hardcoded in `master_orchestrator.py` (not just env vars) | Centralize all config in a single config file or environment variables |
| Dependency management | requirements.txt with minimum version pins (`>=`) | Open version ranges can cause breaking changes | Pin exact versions, use pip-compile or Poetry |
| Scalability | Single-site, single-property design | All scripts hardcoded to `https://www.cim.org/` and GA4 property `341629008` | Abstract site configuration to support multiple properties |
| Maintainability | Good — REPORT_DESIGN_PRINCIPLES.md documents all patterns | 20+ markdown documentation files at root level create clutter | Move docs to `docs/` directory (already started) |
| Security | Secrets in GitHub Actions Secrets; `.gitignore` excludes key files | `gsc-key.json` written to disk during CI run | Use keyless auth; ensure key is deleted after use |
| Monitoring | None | No observability into pipeline health | Add GitHub Actions status badges, failure notifications |

## 19. Final Assessment

### What the project is today

A well-engineered, single-organization SEO automation platform that successfully automates the most time-consuming parts of weekly and monthly SEO reporting for CIM. It pulls data from 4 live APIs (GSC, GA4, PageSpeed, Groq), crawls the CIM website, generates 30+ charts, writes AI-powered executive summaries, and delivers reports to Monday.com and GitHub Pages — all on a cron schedule. The code quality is high, the design system is consistent and well-documented, and the AI integration is thoughtfully implemented with graceful fallbacks.

### What it could become

With targeted additions — primarily conversion tracking, historical trend analysis, anomaly alerting, and email delivery — this could become a genuinely autonomous SEO intelligence system that proactively surfaces opportunities and risks without requiring any manual intervention. The AI infrastructure is already in place; it just needs richer data inputs and more sophisticated prompting.

### The strongest opportunity

**Content decay detection + automated action plans.** The Google Sheets historical data infrastructure is already being built (append_to_sheet is called in every script). Reading that data back to detect decaying pages, then using the existing Groq integration to generate specific refresh recommendations, would create a closed-loop content intelligence system that no off-the-shelf tool provides in this form.

### The biggest risk

**No conversion data.** Every report in this system measures traffic and engagement, but none of it is connected to CIM's actual business outcomes (event registrations, membership sign-ups, course enrollments). Without this connection, the project cannot demonstrate ROI to leadership and will always be perceived as a "nice to have" rather than a business-critical system.

### The highest-value next step

Add GA4 conversion events to the weekly and monthly reports. This is a low-effort change (add conversion metrics to the `_fetch_summary()` call in `ga4_weekly_report.py`) but transforms the entire reporting narrative from "here is our traffic" to "here is how organic search is contributing to CIM's goals."

### The most realistic development path

1. **Week 1**: Add GA4 conversions + email delivery + Slack alerts. These three changes have the highest impact-to-effort ratio.
2. **Month 1**: Build historical trend charts from Google Sheets data + expand keyword tracking to 200+ keywords + add YoY comparison to monthly report.
3. **Quarter 1**: Add content decay detection + anomaly alerting + AI-generated action plans.
4. **Quarter 2**: Add interactive Plotly charts + per-page GSC/GA4 merged view + cannibalization detection.
5. **Year 1**: Multi-site support + revenue attribution + competitor monitoring.
